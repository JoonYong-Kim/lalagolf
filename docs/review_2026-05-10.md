## 2. 우선순위별 발견사항

### P0 (즉시 수정)

1. **팔로우 요청자가 스스로 `accepted` 처리 가능** — `v2/api/app/services/social.py:210-219`
   - `update_follow()`가 `viewer.id in {follower_id, following_id}`만 확인한 뒤 곧바로 임의 `status`를 반영합니다.
   - 영향: A가 B에게 follow 요청을 만든 뒤, A 본인이 `PATCH /api/v1/follows/{A}/{B}`에 `{"status":"accepted"}`를 보내면 B의 `followers` 공개 라운드 접근 권한을 얻을 수 있습니다.
   - 같은 흐름에서 `blocked` 상태도 요청자가 다시 `pending`으로 되돌릴 수 있어 차단 의미가 약해집니다.
   - 권장: `accepted`/`blocked` 전환은 `following_id` 소유자만 허용하고, 요청 취소/재요청은 `follower_id` 소유자 권한으로 분리하세요. 테스트에는 “요청자가 자기 요청을 accepted로 바꾸면 403/400” 케이스를 추가해야 합니다.

2. **코스명 마스킹 정책 변경 필요** — `v2/api/app/services/shares.py:142-146`, social feed serializers
   - 추가 요구사항에 따라 공유/공개 라운드에서 코스명은 숨기지 않아도 됩니다.
   - 기존 설계의 `share_course == false` → `"Shared course"` 마스킹은 더 이상 제품 정책과 맞지 않습니다.
   - 권장: 공유 응답과 소셜 피드에서는 코스명을 항상 표시하고, `share_course`는 제거하거나 레거시 필드로만 남기세요. 정확한 날짜는 여전히 `share_exact_date`로 제어합니다.

### P1 (수정 권장)

1. **라운드 정확한 날짜 공개 옵션 API 스키마 누락** — `v2/api/app/schemas/round.py:78-86`, `v2/api/app/services/rounds.py:100-102`
   - 서비스는 `share_exact_date` 갱신을 지원하지만 `RoundUpdateRequest`에는 해당 필드가 없습니다.
   - 영향: 업로드 commit 시 선택한 정확한 날짜 공개 여부를 라운드 생성 후 일반 PATCH API로 조정할 수 없습니다.
   - 권장: `RoundUpdateRequest`와 `RoundDetailResponse`/프론트 타입에 `share_exact_date`를 추가하고, 공개/링크 공유 UI에서 명시적으로 조정하게 하세요. 코스명은 공개 라운드에서 항상 표시하는 정책으로 정리합니다.

2. **분석 잡 inline 폴백 부재** — `app/services/analysis_jobs.py:172-190`
   - `_try_enqueue_rq_job`은 `settings.analysis_enqueue_enabled`가 False면 그냥 `None` 반환하고 종료. 이때 `AnalysisJob` 행은 `status="queued"`로 남고, `Round.computed_status`는 `pending`에서 영원히 멈춥니다.
   - 기본값(`config.py:14-17`)이 False라서, docker compose를 거치지 않고 로컬 uvicorn으로 띄우면 업로드/수정 후 분석이 절대 돌지 않습니다 (테스트는 `run_analysis_job_in_session`을 직접 호출해서 통과).
   - 권장: (a) `analysis_enqueue_enabled` 기본값을 True로 바꾸거나, (b) enqueue 실패 시 동기 inline 실행 폴백 추가, 둘 중 하나. 운영 환경에는 영향이 없지만 신규 개발자 온보딩에서 가장 빠지기 쉬운 함정.

3. **계획서 vs 실제 잡 함수 정합성** — `app/services/analytics.py:86,92,101`
   - `recalculate_user_expected_table`, `recalculate_shot_values`, `generate_insights` 세 함수는 실제로 어디서도 호출되지 않습니다 (`grep` 0건). 실제 재계산 체인은 `recalculate_round_metrics`(analytics.py:54) 안에서 `_recalculate_round_prior_expected_table` → `_replace_shot_values` → `_replace_insights` 순서로 한 번에 처리.
   - 권장: 셋 중 하나만 선택 — (a) 진짜 admin/배치용으로 라우트에 노출하거나, (b) 사용 안 함을 주석으로 명시하거나, (c) 코드를 제거해서 dead code를 줄이세요. P5.2의 "Jobs are idempotent" 검증 시 혼동을 줍니다.

4. **Ruff 검사 실패** — `v2/api/alembic/versions/20260510_0011_social_visibility.py`, `v2/api/app/api/v1/auth.py`, 기타 import 정렬
   - `cd v2/api && ruff check .` 결과 9건 실패. 긴 줄 2건과 import 정렬 7건입니다.
   - 테스트는 통과하지만 CI에 ruff가 포함되면 현재 상태로 실패합니다.
   - 권장: `ruff check --fix .`로 import 정렬을 먼저 정리하고, 마이그레이션 긴 줄 2건은 수동 줄바꿈하세요.

### P2 (UX/완성도)
